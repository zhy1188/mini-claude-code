# 状态流转机制：FSM + 4 层管理

## 一、概述

NexusAgent 的状态管理采用 **有限状态机（FSM）+ 3 层辅助机制**，共 4 层，协同管理 Agent 从接收用户输入到完成回复的完整生命周期。

### 为什么用 FSM，而不是裸枚举

早期用 `AgentState` 裸枚举（`idle`, `gathering`, `thinking`...）配合 `if-else` 判断状态，问题在于：
- **没有约束**：任何代码都能把状态设成任意值，`idle` 可以直接跳到 `acting`，跳过了中间步骤
- **难以排查**：状态出 bug 时无法定位是哪里非法转换的
- **扩展困难**：加一个新状态要手动改所有判断逻辑

FSM 的核心改进是 **用一张转换表定义所有合法路径**，转换时查表验证，非法转换直接抛异常。

### 4 层状态管理架构

```
第 1 层 ┃ StateMachine (FSM)      定义"状态是什么"，强制转换合法性
第 2 层 ┃ Checkpoint               在关键节点持久化状态，崩溃后可恢复
第 3 层 ┃ asyncio.Event (取消)     用户按 Ctrl+C 时安全中断
第 4 层 ┃ ToolTracker              追踪每个工具的生命周期（pending→running→done）
```

## 二、核心组件详解

### 2.1 状态转换表（`state_machine.py:22-31`）

```python
VALID_TRANSITIONS: dict[str, set[str]] = {
    "idle":      {"gathering"},       # 空闲 → 只能进入收集阶段
    "gathering": {"thinking", "compact", "done"},
    "thinking":  {"acting", "verifying", "done", "error"},
    "acting":    {"gathering", "thinking", "verifying", "error"},
    "verifying": {"gathering", "done", "error"},
    "compact":   {"thinking", "error"},
    "error":     {"idle"},
    "done":      {"idle"},
}
```

**关键设计**：

| 转换 | 含义 | 举例 |
|------|------|------|
| `idle → gathering` | 用户输入触发，开始一轮对话 | 用户发消息，Agent 开始处理 |
| `gathering → thinking` | 上下文收集完，开始调 LLM | 正常路径 |
| `gathering → compact` | 上下文超限，先压缩再思考 | Token 快满时走这条 |
| `gathering → done` | 钩子拦截或取消，直接结束 | Hook 返回 blocked |
| `thinking → acting` | LLM 返回了工具调用 | Agent 说"我要读文件" |
| `thinking → verifying` | LLM 返回了纯文本 | Agent 直接回复用户 |
| `acting → gathering` | 工具执行完，重新收集上下文再调 LLM | 工具结果作为消息加回去 |
| `acting → thinking` | 多个工具批量调用后继续思考 | 连续工具调用的中间态 |
| `compact → thinking` | 压缩完回到思考阶段 | 压缩后继续调 LLM |

### 2.2 StateMachine 类（`state_machine.py:34-84`）

```python
@dataclass
class StateMachine:
    current: str = "idle"                              # 当前状态
    _callbacks: dict[str, list[Callable]] = field()    # 状态回调表
```

**三个核心方法**：

```python
# 1. transition() — 正常转换（会验证）
def transition(self, target):
    allowed = VALID_TRANSITIONS.get(self.current, set())
    if target not in allowed:
        raise InvalidStateTransition(self.current, target)
    self.current = target
    # 触发回调
    for cb in self._callbacks.get(target, []):
        cb(old, target)

# 2. force() — 强制转换（跳过验证，用于异常恢复）
def force(self, target):
    self.current = target

# 3. can_transition() — 预判合法性（不实际转换）
def can_transition(self, target) -> bool:
    return target in VALID_TRANSITIONS.get(self.current, set())
```

### 2.3 Legacy 枚举兼容层（`master.py:80-81, 116-128`）

项目保留了旧的 `AgentState` 枚举（定义在 `models.py:13`），通过 `_sync_legacy_state()` 同步：

```python
def _sync_legacy_state(self) -> None:
    """将 FSM 状态同步到旧版 AgentState 枚举（向后兼容）"""
    state_map = {
        "idle": AgentState.IDLE,
        "gathering": AgentState.GATHERING,
        "thinking": AgentState.THINKING,
        "acting": AgentState.ACTING,
        "verifying": AgentState.VERIFYING,
        "compact": AgentState.COMPACTING,
        "done": AgentState.DONE,
        "error": AgentState.ERROR,
    }
    self.state = state_map.get(self.state_machine.current, AgentState.IDLE)
```

**为什么要同步？** TUI、外部接口或旧代码可能还在用 `agent.state`（枚举值），同步确保新旧两套状态始终一致。未来可逐步移除这个兼容层。

## 三、一轮完整的状态流转

以用户问"帮我修 auth.py 的 bug"为例，完整追踪所有状态转换：

### 3.1 正常路径（无压缩，无工具调用）

```
idle → gathering → thinking → verifying → done → idle

  idle ───────────────────────────────────────────── 启动，等待输入
  ↓ [用户输入: "修 auth.py 的 bug"]
  gathering ──────────────────────────────────────── 收集上下文
    ├─ 从输入中提取 "auth.py" 路径
    ├─ 预读 auth.py 文件内容，注入上下文
    ├─ 添加 UserMessage 到历史
    └─ 保存检查点（user_input 阶段）
  ↓
  thinking ───────────────────────────────────────── 调用 LLM
    ├─ 保存检查点（before_llm 阶段）
    ├─ 流式调用 LLM，token 逐字输出到 TUI
    └─ LLM 返回纯文本回复（不涉及工具）
  ↓
  verifying ──────────────────────────────────────── 验证结果
    ├─ 添加 AssistantMessage 到历史
    ├─ 触发 post_response 钩子
    └─ 保存会话
  ↓
  done ───────────────────────────────────────────── 本轮结束
    ├─ 清除检查点
    └─ 等待下一个用户输入
```

### 3.2 带工具调用的路径（最常见）

```
idle → gathering → thinking → acting → gathering → thinking → verifying → done

  idle
  ↓
  gathering ─── 收集上下文
  ↓
  thinking ─── LLM 返回: "我需要先读 auth.py" + ToolCall(Read)
  ↓
  acting ───── 执行 Read 工具
    ├─ 读取 auth.py 内容
    ├─ 添加 ToolResultMessage 到历史
    └─ 保存检查点（tool_done 阶段）
  ↓
  gathering ── 工具结果已加入历史，重新进入循环
  ↓
  thinking ─── 带工具结果的上下文再次调 LLM
    └─ LLM 返回: "我已发现 bug，需要修改" + ToolCall(Write)
  ↓
  acting ───── 执行 Write 工具
    ├─ 权限检查（Write 是 "ask"）
    ├─ 用户确认后执行写入
    └─ 保存检查点
  ↓
  gathering ── 再次循环
  ↓
  thinking ─── LLM 返回纯文本："已修复，问题是..."
  ↓
  verifying ─ 验证结果，添加回复
  ↓
  done
```

### 3.3 带压缩的路径

```
idle → gathering → compact → thinking → acting → thinking → verifying → done

  idle
  ↓
  gathering ─── 收集上下文
    └─ 检查 needs_compaction(): total_tokens > 200000 * 0.75 = 150000
    └─ 条件满足，进入压缩阶段
  ↓
  compact ──── 压缩上下文
    ├─ 判断压缩阶段（基于 token 比率）:
    │   ratio < 0.85  → 阶段 1: 移除最旧的非关键文本
    │   0.85-0.95     → 阶段 2: LLM 摘要旧消息
    │   >= 0.95       → 阶段 3: 极端压缩，只保留摘要+2条最新
    ├─ 执行压缩，替换 messages 列表
    └─ 回到 thinking
  ↓
  thinking ─── 压缩后的精简上下文调 LLM
  ...
```

### 3.4 用户中断路径

```
idle → gathering → thinking →(Ctrl+C)→ done → idle

  idle
  ↓
  gathering
  ↓
  thinking ─── 流式输出中，用户按 Ctrl+C
    └─ _cancel_event.set()
    └─ 流式循环检测到 _cancel_event.is_set()
    └─ _graceful_stop():
        ├─ state_machine.force("done")    ← 强制，不验证转换表
        ├─ _sync_legacy_state()
        └─ _save_session()                ← 保存当前对话
        └─ 保留检查点（用户可以 /resume）
  ↓
  done ───── 优雅停止
```

## 四、关键实现细节

### 4.1 为什么每次转换都用 try/except + force 兜底

看 `master.py:157-161` 的代码：

```python
try:
    self.state_machine.transition("gathering")
except Exception:
    self.state_machine.force("gathering")
```

**原因**：这是一个 **防御性编程** 模式。理论上如果从合法状态转换，`transition()` 不会抛异常。但在某些边缘情况下（比如恢复后的状态不明确、hook 异常退出后等），FSM 可能处于不期望的状态。用 `force()` 兜底确保：
- 正常情况：走 `transition()`，验证合法性，触发回调
- 异常情况：走 `force()`，强制推进，至少不让整个循环卡死

这是一个 **宁可跳过验证也不要阻塞用户** 的权衡。

### 4.2 检查点与状态的配合

检查点在 5 个关键节点保存状态（`master.py` 中）：

| 检查点时机 | 代码位置 | 保存的信息 |
|-----------|---------|-----------|
| 用户输入后 | line 190 | 当前状态= gathering, 用户输入文本 |
| LLM 调用前 | line 218 | 当前状态= thinking, 完整消息历史 |
| 工具执行后 | line 332 | 当前状态= acting, 工具追踪状态 |
| 取消时 | line 146 | 当前状态= done, 保留恢复能力 |
| 完成后 | line 378 | 清除所有检查点 |

**恢复流程**（`master.py:404-451`）：
1. `resume()` 加载最新检查点
2. 恢复 session_id
3. 按 role 重建消息历史（UserMessage / AssistantMessage / SystemMessage）
4. 重新计算 token 总数
5. 继续执行

### 4.3 工具生命周期追踪（第 4 层）

`ToolTracker` 追踪每个工具调用的生命周期状态：

```
pending → running → done/failed/cancelled
```

在 `master.py:265-309` 中：
```python
# 注册所有工具（初始 pending）
for tc in tool_calls:
    self.tool_tracker.create(tc.id, tc.name, tc.input)

for tc in tool_calls:
    self.tool_tracker.start(tc.id)      # → running
    # ... 执行工具 ...
    if result.is_error:
        self.tool_tracker.fail(tc.id, error)  # → failed
    else:
        self.tool_tracker.complete(tc.id, output)  # → done
```

**为什么分两步注册？** 先全部 `create`（pending），再逐个 `start`（running）。这是为了支持 **取消机制**——如果用户在中间取消，`pending()` 方法能找出所有还没开始执行的工具并标记为 `cancelled`。

### 4.4 取消机制的 4 个检测点

`_check_cancelled()` 和 `_cancel_event.is_set()` 在 4 个位置检查：

```
1. 钩子拦截检查（line 176）—— 在处理用户输入之前
2. 循环边界（line 197）—— 每次循环开始前
3. 流式输出中（line 236）—— token 逐字到达时
4. 每个工具执行前（line 270）—— 工具调用之间
```

这确保了无论 Agent 处于哪个阶段，用户按 Ctrl+C 都能在合理时间内响应。
