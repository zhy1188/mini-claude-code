# NexusAgent 核心改进技术报告

> 本文档记录 4 个核心改进的设计思路、实现过程和面试讲述要点。

---

## 一、Sub-Agent 接入主循环

### 1.1 改进前的问题

**现状**：MasterAgent 主循环是单线程的，只能串行处理任务。当用户说"帮我分析这三个文件并运行测试"时，Agent 需要依次读每个文件、再运行测试——上下文窗口被大量工具输出填满，效率低。

**本质问题**：
- 单线程主循环无法利用 LLM 的上下文隔离特性
- 可并行的子任务（如读不同文件、跑不同测试套件）被串行执行
- 所有工具输出都挤在一个上下文窗口中

### 1.2 设计方案

**核心思路**：引入一个 `Task` 工具，让主 Agent 可以派生独立的子 Agent，每个子 Agent 有自己隔离的上下文窗口和受限的工具集，并发执行后聚合结果。

**架构层次**：
```
MasterAgent (主循环)
    ├── 识别可并行任务
    ├── 调用 Task 工具
    │       └── AgentOrchestrator (编排器)
    │               ├── SubAgent A: 分析 auth.py
    │               ├── SubAgent B: 分析 db.py
    │               └── SubAgent C: 运行测试
    └── 聚合结果 → 继续主循环
```

### 1.3 实现过程

**Step 1: 完善 Orchestrator 的子 Agent 类型**

```python
# agent/orchestrator.py
_SUBAGENT_PROMPTS = {
    "general": "你是一个专注的编码助手，高效完成任务并返回摘要。",
    "research": "你是一个研究助手，分析代码库，阅读文件，搜索模式，提供详细报告。不修改文件。",
    "code": "你是一个代码修改助手，仔细读文件后再修改，确保代码能运行。",
}
```

关键设计：不同类型的子 Agent 有不同的 System Prompt 和工具集，实现**能力隔离**。

**Step 2: 增强 SubTask 支持**

```python
class SubTask:
    def __init__(
        self,
        description: str,
        tool_names: list[str] | None = None,
        subagent_type: str = "general",
        prompt: str = "",
    ):
        self.description = description          # 子任务描述
        self.tool_names = tool_names            # 可用工具集
        self.subagent_type = subagent_type      # 子 Agent 类型
        self.prompt = prompt                    # 自定义指令
```

**Step 3: 实现 Task 工具**

```python
# tools/builtin/task.py
class TaskTool(Tool):
    name = "Task"
    description = "启动子Agent处理复杂的独立任务。适用于可并行执行的任务。"
    parameters = {
        "description": {"type": "string", "description": "子任务描述", "required": True},
        "subagent_type": {"type": "string", "description": "类型: general/research/code"},
        "prompt": {"type": "string", "description": "详细指令"},
    }

    def set_orchestrator(self, orchestrator, all_tools):
        """延迟绑定编排器，避免循环依赖。"""
        self._orchestrator = orchestrator
        self._all_tools = all_tools

    async def execute(self, description, subagent_type="general", prompt=""):
        task = SubTask(description=description, prompt=prompt)
        results = await self._orchestrator.spawn([task])
        return ToolResult(content=f"子Agent完成:\n{results[0].summary}")
```

**关键设计决策——为什么用延迟绑定？**

TaskTool 需要引用 AgentOrchestrator，但 AgentOrchestrator 又在 MasterAgent 的 `__init__` 中创建，而 MasterAgent 需要 TaskTool 已经在 registry 里。这是一个典型的**循环依赖**问题。

解决方案：
1. 先创建 ToolRegistry 并注册 TaskTool（此时 orchestrator 还不存在）
2. 创建 MasterAgent，在 `__init__` 中创建 orchestrator
3. 调用 `_wire_task_tool()` 将 orchestrator 注入到 TaskTool

```python
# agent/master.py
def __init__(self, ...):
    self.orchestrator = AgentOrchestrator(self.llm, self.registry)
    self._wire_task_tool()

def _wire_task_tool(self):
    task = self.registry.get("Task")
    if isinstance(task, TaskTool):
        task.set_orchestrator(self.orchestrator, self.registry)
```

**Step 4: 接入主循环**

在 `main.py` 中注册 TaskTool：
```python
registry.register(TaskTool(workdir))
agent = MasterAgent(..., workdir=workdir)
```

MasterAgent 的 `run()` 循环不需要任何改动——因为 Task 工具和其他工具一样遵循 `ToolCall → Execute → Result` 的标准流程。这正是**工具抽象**的价值：主循环不关心工具内部做什么，只管调用和收集结果。

### 1.4 关键技术点

| 技术点 | 说明 |
|--------|------|
| **循环依赖解耦** | 延迟绑定模式（set_orchestrator） |
| **上下文隔离** | 每个 SubAgent 有独立的 ContextManager |
| **并发执行** | `asyncio.gather(*[agent.run() for agent in agents])` |
| **工具路由** | 根据 subagent_type 分配不同工具集 |
| **错误隔离** | `return_exceptions=True` 单个失败不影响其他 |

### 1.5 面试讲述要点

> "我在单线程 Agent 循环中加入了子 Agent 编排能力。核心设计是 Task 工具 + Orchestrator 两层架构：主 Agent 通过 Task 工具派发子任务，Orchestrator 根据任务类型（research/code/general）创建具有不同 System Prompt 和工具集的 SubAgent，用 asyncio.gather 并发执行。

> 技术上最大的挑战是循环依赖：TaskTool 需要 Orchestrator，但 Orchestrator 需要 ToolRegistry，而 ToolRegistry 需要 TaskTool。我用延迟绑定模式解决：先注册工具，再创建 Orchestrator，最后注入引用。

> 这个设计的价值在于**可扩展性**——新增子 Agent 类型只需要在 `_SUBAGENT_PROMPTS` 里加一行配置，不需要改动主循环代码。"

## 二、Hooks 注入主循环

### 2.1 改进前的问题

**现状**：HookEngine 和 HookConfig 都写好了，但没有接入任何生命周期节点。用户无法在工具执行前后添加自定义行为。

**本质问题**：缺乏扩展点。如果用户想审计所有 Bash 命令、或记录工具执行时间、或在 LLM 响应前做安全检查，需要改源码。

### 2.2 设计方案

**核心思路**：在 MasterAgent 的 `run()` 方法的关键节点插入 Hook 触发点，让 HookEngine 可以执行用户配置的钩子命令。

**钩子生命周期**：
```
用户输入
  ↓
PRE_USER_MESSAGE   ← 新增：可以在处理用户输入前做预处理
处理输入
  ↓
POST_USER_MESSAGE  ← 新增：用户输入处理完后触发
构建消息 → LLM → 解析响应
  ↓
工具循环:
  对每个 ToolCall:
    PRE_TOOL_USE   ← 新增：工具执行前拦截
    权限检查 → 执行工具
    POST_TOOL_USE  ← 新增：工具执行后记录
  ↓
LLM 文本响应
  ↓
POST_RESPONSE      ← 新增：响应发送前/后处理
```

### 2.3 实现过程

**Step 1: 添加 HookEngine 依赖**

```python
# agent/master.py
def __init__(self, ..., hook_engine: HookEngine | None = None):
    ...
    self.hook_engine = hook_engine
```

HookEngine 是可选的。如果没传，所有 hook 检查直接跳过，零性能开销。

**Step 2: PRE_USER_MESSAGE 钩子**

```python
async def run(self, user_input: str):
    # 触发前置钩子
    if self.hook_engine:
        hook_result = await self.hook_engine.trigger(
            HookType.PRE_USER_MESSAGE,
            {"user_input": user_input}
        )
        if hook_result.blocked:
            await self.tui.show_status(f"Hook blocked: {hook_result.reason}")
            self.state = AgentState.DONE
            return
```

关键设计：阻塞型钩子可以**阻止用户输入被处理**。这实现了安全审查——比如用户输入包含敏感词时，钩子可以返回 blocked=True，Agent 直接拒绝处理。

**Step 3: PRE/POST_TOOL_USE 钩子**

```python
for tc in tool_calls:
    # PRE: 工具执行前
    if self.hook_engine:
        hook_ctx = {
            "tool_name": tc.name,
            "tool_input": json.dumps(tc.input),
        }
        hook_result = await self.hook_engine.trigger(
            HookType.PRE_TOOL_USE, hook_ctx
        )
        if hook_result.blocked:
            await self.tui.show_status(f"Hook blocked {tc.name}")
            continue  # 跳过这个工具，继续下一个

    # 权限检查 → 执行工具
    result = await self._execute_tool(tc)

    # POST: 工具执行后
    if self.hook_engine:
        await self.hook_engine.trigger(
            HookType.POST_TOOL_USE,
            {
                "tool_name": tc.name,
                "tool_input": json.dumps(tc.input),
                "tool_output": result.content,
                "status": status,
            },
        )
```

关键设计：
- POST 钩子包含**工具输出**，可以做日志记录、结果验证
- PRE 钩子阻塞时 `continue` 而不是 `break`，保证其他工具继续执行

**Step 4: POST_RESPONSE 和 POST_USER_MESSAGE 钩子**

```python
# LLM 响应完成后
if full_content and self.hook_engine:
    await self.hook_engine.trigger(
        HookType.POST_RESPONSE,
        {"response": full_content},
    )

# 整个 turn 结束后
if self.hook_engine:
    await self.hook_engine.trigger(
        HookType.POST_USER_MESSAGE,
        {"user_input": user_input, "state": self.state.value},
    )
```

### 2.4 关键技术点

| 技术点 | 说明 |
|--------|------|
| **可选依赖** | HookEngine 为 None 时零开销，不影响主流程 |
| **阻塞传播** | 钩子返回 `blocked=True` 可以阻止后续操作 |
| **上下文传递** | 每个钩点传递不同的 context dict，钩子命令可用 `${tool_name}` 插值 |
| **非隔离失败** | 钩子执行失败不影响主循环（在 HookEngine 内部 try/except） |

### 2.5 面试讲述要点

> "我实现了类似 Web 框架中间件和 Git hooks 的生命周期钩子系统。在 Agent 的 5 个关键节点（pre/post user message, pre/post tool use, post response）注入触发点。

> 技术上有三个设计细节：
> 1. **HookEngine 是可选依赖**——不配置时 `if self.hook_engine` 直接跳过，零性能损失
> 2. **阻塞型钩子可以阻止操作**——pre_tool_use 钩子返回 blocked=True 时跳过该工具，但 `continue` 保证其他工具继续执行
> 3. **上下文插值**——钩子命令支持 `${tool_name}`, `${tool_output}` 等变量，类似 Shell 模板引擎

> 类比的话，这和 Express 的 middleware、Spring 的 AOP、Git hooks 是同一个模式：在核心流程的关键节点预留扩展点，让用户通过配置而非改源码来定制行为。"

## 三、记忆系统（读写双向 + 两级索引）

### 3.1 改进前的问题

**现状**：MemorySystem 类写好了，支持 user/feedback/project/reference 四种记忆类型，但只能读不能写。LLM 在对话中识别到用户偏好或项目决策时，无法保存下来。

**本质问题**：
- 记忆系统是"空壳"——有目录结构、有读取逻辑，但没有任何写入入口
- 每次对话结束后，用户的纠正、项目决策全部丢失
- 类比 Claude Code：它有 auto memory 机制，Agent 可以主动调用工具保存记忆

### 3.2 设计方案

**核心思路**：对标 Claude Code 的 auto memory——将记忆写入封装为 LLM 可调用的工具（MemoryWriteTool），让 Agent 自行识别记忆点并保存。采用**两级索引**结构：`MEMORY.md` 指针索引 + `.md` 文件（带 YAML frontmatter 元数据）。

**存储结构**：
```
.nexus/
├── MEMORY.md              # 自动生成的索引文件
└── memory/
    ├── user.md            # 用户偏好（frontmatter + 正文）
    ├── feedback.md        # 纠正与指导
    ├── project.md         # 项目决策
    └── reference.md       # 外部引用
```

**记忆文件格式（YAML frontmatter）**：
```markdown
---
name: 测试偏好
type: user
description: 用户偏好的测试框架
created: 2026-04-18T10:30:00Z
---

项目使用 pytest，不使用 unittest。优先使用 fixtures 而非 setup/teardown。
```

### 3.3 实现过程

**Step 1: 轻量级 frontmatter 解析器（无外部依赖）**

```python
# memory/frontmatter.py
def parse_frontmatter(text: str) -> tuple[dict, str]:
    """解析 YAML frontmatter，返回 (元数据, 正文)。"""
    pattern = r"^---\s*\n(.*?)\n---\s*\n(.*)$"
    match = re.match(pattern, text, re.DOTALL)
```

为什么不用 PyYAML？引入外部依赖增加安装复杂度。frontmatter 格式极其简单（只有 key: value 行），用正则足够。

**Step 2: MemoryIndex 索引管理器**

```python
# memory/index.py
class MemoryIndex:
    def add_entry(self, entry) -> bool:    # 新增，自动去重
    def update_entry(self, name, type, **updates) -> bool
    def remove_entry(self, name, type) -> bool
    def list_entries(self, type=None) -> list
    def save(self) -> None                  # 渲染 MEMORY.md
```

解析 MEMORY.md 时使用正则匹配行格式：`- [name](path) -- description`，按 `## type` 分段。

**Step 3: MemoryWriteTool（核心工具）**

```python
# tools/builtin/memory_write.py
class MemoryWriteTool(Tool):
    name = "MemoryWrite"
    description = "Save, update, forget, or list cross-session memories."
    parameters = {
        "operation": {"enum": ["save", "update", "forget", "list"]},
        "name": {"type": "string"},
        "memory_type": {"enum": ["user", "feedback", "project", "reference"]},
        "content": {"type": "string"},
        "description": {"type": "string"},
    }
```

一个工具四种操作：
- **save**：验证参数 → 检查 name+type 去重 → 写文件（带 frontmatter）→ 更新索引
- **forget**：从索引删除 → 标准文件清空内容，动态文件直接删除
- **list**：按类型分组展示
- 验证规则：name 非空、type 合法、content ≤10000 字符

**Step 4: 注册到 MasterAgent 循环**

```python
# agent/master.py
nexus_dir = self.workdir / ".nexus"
self.memory_write_tool = MemoryWriteTool(nexus_dir)
self.registry.register(self.memory_write_tool)  # LLM 可见

# 主循环结束时自动保存会话
self.session_manager.save(self._current_session_id, messages)
```

**Step 5: 权限自动批准**

```python
# permission/policy.py
"MemoryWrite": "approve",  # 不每次询问用户
"SessionSave": "approve",
```

### 3.4 完整写入流程

```
用户: "这个项目的测试都用 pytest，不用 unittest"
    ↓
[LLM] 识别到用户偏好 → 决定调用 MemoryWrite 工具
    ↓
[LLM] operation=save, name="测试偏好", type=user, content="项目使用 pytest"
    ↓
[PermissionGate] MemoryWrite=approve → 通过
    ↓
[MemoryWriteTool] 验证 → 写 .nexus/memory/user.md → 更新 MEMORY.md 索引
    ↓
[LLM] 收到成功结果 → 回复用户
    ↓
[下次启动] 自动加载记忆 → 注入 system_prompt → Agent 记住了
```

### 3.5 关键技术点

| 技术点 | 说明 |
|--------|------|
| **LLM 驱动写入** | Agent 自行判断何时保存，非 Hook 被动触发 |
| **两级索引** | MEMORY.md 指针 + .md 文件，快速加载/查询 |
| **Frontmatter 元数据** | name/type/description/created，支持结构化查询 |
| **去重更新** | 同 name+type 视为更新，不产生重复条目 |
| **无外部依赖** | 纯正则解析 frontmatter，不依赖 PyYAML |
| **标准文件保护** | user.md 等标准文件 forget 时清空而非删除 |

### 3.6 面试讲述要点

> "我实现了完整的记忆写入系统，对标 Claude Code 的 auto memory 机制。
>
> 核心设计是把记忆操作封装为 LLM 可调用的工具——MemoryWriteTool，支持 save/forget/update/list 四种操作。Agent 在对话中识别到用户偏好或项目决策时，主动调用工具保存。
>
> 存储上用了两级索引：MEMORY.md 作为指针索引，每个记忆文件带 YAML frontmatter 元数据。这样启动时只需读索引文件就能知道有哪些记忆，按需加载。
>
> 技术难点在**去重和一致性**：同一 name+type 的重复 save 应该更新而非新建，forget 操作要同时更新索引和文件。我通过 MemoryIndex 类集中管理所有增删改操作，保证索引和文件的一致性。
>
> 这个系统的价值是渐进式学习——用户在对话中纠正 Agent 的行为后，这些纠正被持久化，下次启动时 Agent 自动加载，不会再犯同样的错误。"

## 四、工具结果缓存

### 4.1 改进前的问题

**现状**：Bash 工具每次执行都启动子进程。如果 LLM 重复调用同一个只读命令（如 `git status`、`ls`），每次都产生相同的输出，但消耗了 API 调用的 token 和等待时间。

**本质问题**：没有利用只读命令的**幂等性**。相同命令相同参数的结果是一定的，应该缓存。

### 4.2 设计方案

**核心思路**：在 BashTool 内部维护一个结果缓存。只读命令第一次执行后缓存结果，第二次直接返回。写命令（git commit、make 等）不缓存。

### 4.3 实现过程

**Step 1: 添加缓存存储**

```python
# tools/builtin/bash.py
class BashTool(Tool):
    def __init__(self, workdir, config=None):
        super().__init__(workdir)
        self._cache: dict[str, ToolResult] = {}
```

**Step 2: 生成缓存键**

```python
def _cache_key(self, command: str) -> str:
    """用 MD5 哈希命令内容作为缓存键。"""
    return hashlib.md5(command.encode()).hexdigest()
```

为什么用 MD5 而不是直接用命令字符串？因为命令可能很长、包含特殊字符，用哈希做字典键更安全和高效。

**Step 3: 识别只读命令**

```python
async def execute(self, command, timeout=None, dangerous=False):
    # ... 安全检查 ...

    # 只读判断：只有不包含写操作的命令才缓存
    read_only = all(
        not command.startswith(cmd)
        for cmd in ("git", "make", "python", "pip", "npm", "cargo", "cmake")
    )

    # 检查缓存
    cache_key = self._cache_key(command)
    if read_only and cache_key in self._cache:
        return self._cache[cache_key]
```

关键设计：`git`, `make`, `python` 等命令**不缓存**，因为它们有副作用（`git commit`, `make build`, `python script.py`）。而 `ls`, `cat`, `echo`, `find`, `grep` 等默认命令是只读的，可以缓存。

**Step 4: 写入缓存**

```python
    # 执行命令 ...
    result = ToolResult(content=output, metadata={...})

    # 缓存只读结果
    if read_only:
        self._cache[cache_key] = result

    return result
```

### 4.4 关键技术点

| 技术点 | 说明 |
|--------|------|
| **幂等性利用** | 只读命令结果确定 → 可缓存 |
| **MD5 键** | 命令字符串 → 固定长度哈希，高效字典查找 |
| **副作用隔离** | 写命令不缓存，保证一致性 |
| **内存缓存** | 简单 dict，无持久化，会话结束自动清除 |

### 4.5 面试讲述要点

> "我给 Bash 工具加了结果缓存，利用只读命令的幂等性。用 MD5 哈希命令内容作为缓存键，`ls`、`cat`、`echo` 等命令第一次执行后缓存结果，后续相同命令直接返回。

> 关键是**副作用判断**：`git`、`make`、`python` 等命令有写操作，不缓存。只缓存纯只读命令。这类似于数据库的查询缓存 vs 写操作的区分。

> 这个改进的价值在于减少 token 消耗和等待时间——LLM 经常会重复调用 `git status` 来确认状态，缓存后避免了重复执行。同时缓存是内存级别的，会话结束自动清除，不需要持久化逻辑。"

## 五、Skill 系统（Markdown 指令文件）

### 5.1 改进前的问题

**现状**：Agent 的行为完全由系统提示和工具定义决定。如果想让 Agent 学会一种新的工作流程（如部署、代码审查、数据库迁移），需要改系统提示或写死在代码中。

**本质问题**：
- 缺乏**可插拔的行为扩展机制**——用户无法在不改代码的情况下教 Agent 做新事
- Claude Code 的解决方案是 Skills：`.claude/commands/` 下的 `.md` 文件自动成为可调用指令

### 5.2 设计方案

**核心思路**：对标 Claude Code 2026 年的 Skills 系统——一个 `.md` 文件就是一个 Skill，放在 `.nexus/skills/` 目录下自动被发现，用户通过 `/skill-name` 或自然语言调用。

**架构层次**：
```
.nexus/skills/
├── deploy.md          → /deploy 或 "部署" 触发
├── code-review.md     → /code-review 或 "审查代码" 触发
└── ...

用户输入 → SkillMatcher → SkillExecutor → 指令注入到 Agent prompt
```

### 5.3 实现过程

**Step 1: Skill 数据模型 + Markdown 解析**

```python
# skills/models.py
class Skill(BaseModel):
    name: str
    description: str = ""
    content: str  # Markdown 正文
    source: str   # 文件来源

    @classmethod
    def from_file(cls, path: Path, scope: str = "project") -> "Skill":
        """解析 Markdown 文件，分离 frontmatter 和正文。"""
        content = path.read_text(encoding="utf-8")
        name = path.stem  # 文件名作为 skill 名
        if content.startswith("---"):
            parts = content.split("---", 2)
            frontmatter = parts[1].strip()
            body = parts[2].strip()
            # 解析 name, description, version
        return cls(name=name, description=..., content=body)
```

**Step 2: SkillRegistry 发现 + 加载**

```python
# skills/registry.py
class SkillRegistry:
    def add_directory(self, path: Path, scope: str = "project"):
        self.directories.append((path, scope))

    def scan(self) -> None:
        """扫描所有目录，加载 .md 文件为 skill。"""
        self.skills.clear()
        for directory, scope in sorted(self.directories, key=lambda x: x[1]):
            for md_file in directory.glob("*.md"):
                skill = Skill.from_file(md_file, scope=scope)
                self.skills[skill.name] = skill  # project 覆盖 global
```

关键设计：项目级 skill 覆盖全局同名 skill，通过 `sorted` 按 scope 排序保证优先级。

**Step 4: SkillExecutor 指令注入**

```

{skill.content}
```

LLM 按 skill 中定义的步骤调用工具执行。

**Step 5: 接入 main.py 和 REPL**

```python

skill_registry.add_directory(workdir / ".nexus" / "skills", scope="project")
skill_registry.add_directory(Path.home() / ".nexus" / "skills", scope="global")
skill_registry.scan()

# 注册每个 skill 为 slash command
for skill in skill_registry.list_skills():
    tui.register_command(skill.name, skill_handler, skill.description)

# REPL 中拦截输入，匹配 skill
enhanced, matched = skill_executor.process_input(user_input)
if matched:
    await agent.run(enhanced)  # 使用 skill 注入的 prompt
```

### 5.4 关键技术点

| 技术点 | 说明 |
|--------|------|
| **Markdown 即 Skill** | 无需新 DSL，纯 Markdown + frontmatter |
| **两级作用域** | 全局 `~/.nexus/skills/` + 项目 `.nexus/skills/` |
| **三级匹配** | Slash → 名称关键词 → 描述关键词，渐进式覆盖 |
| **指令注入** | Skill 内容作为 prompt 前置，不改变 Agent 循环逻辑 |
| **自动注册** | 每个 skill 自动成为 `/skill-name` slash command |

> 技术架构分三层：
> 1. **SkillRegistry**：扫描目录，解析 frontmatter 和正文，建立索引
> 2. **SkillMatcher**：三级匹配——精确 slash → 名称关键词 → 描述关键词
> 3. **SkillExecutor**：将 skill 指令注入到 Agent prompt 中，让 LLM 按步骤执行

> 这个设计的核心思想是 **Context Engineering**：Skill 本质上是一种结构化的上下文注入方式。它不是写死在代码里的逻辑，而是用自然语言教 Agent 如何做一件事。核心思想与 Claude Code 2026 年的 Skills 系统完全一致。"
