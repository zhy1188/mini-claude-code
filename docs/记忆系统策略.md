# 记忆系统：跨会话记忆与单轮会话的双层设计

## 一、概述

NexusAgent 的记忆系统分为两个独立但协同的层次：

1. **跨会话记忆（Cross-Session Memory）**：持久化存储用户偏好、项目决策、经验教训等，在多次对话之间持久保留。类比 Claude Code 的 `.claude/` 记忆目录。
2. **单轮会话（Session）**：保存每次对话的完整消息历史（JSON），用于查看历史记录和断点恢复。

### 为什么需要两层记忆

| 维度 | 跨会话记忆 | 单轮会话 |
|------|-----------|---------|
| **存储内容** | 用户偏好、经验教训、项目决策 | 完整对话消息（User/Assistant/Tool 消息） |
| **生命周期** | 长期存在，跨越多轮对话 | 每轮对话一个，完成后归档 |
| **作用** | 注入系统提示，指导 Agent 行为 | 恢复中断的对话，回顾历史 |
| **类比** | Claude Code 的 `.claude/` | Claude Code 的 `.claude/sessions/` |
| **格式** | Markdown + YAML frontmatter | JSON（序列化的消息列表） |

### 记忆系统的整体架构

```
.nexus/
├── memory/                    ← 跨会话记忆存储
│   ├── user.md               用户偏好、角色、知识
│   ├── feedback.md           用户的指导和纠正
│   ├── project.md            项目目标、决策、状态
│   ├── reference.md          外部系统指针
│   └── (其他自定义 .md)       动态创建的记忆文件
├── MEMORY.md                  ← 记忆索引（自动维护）
├── checkpoints/               ← 检查点文件
├── sessions/                  ← 单轮会话 JSON
└── skills/                    ← 技能定义
```

## 二、跨会话记忆详解

### 2.1 四级记忆分类

定义在 `memory/memory.py:25`：

```python
MEMORY_TYPES = ["user", "feedback", "project", "reference"]
```

| 类型 | 文件 | 内容 | 举例 |
|------|------|------|------|
| **user** | `user.md` | 用户身份、角色、技术栈、工作习惯 | "用户是 Python 后端工程师，熟悉 Go，不熟悉前端" |
| **feedback** | `feedback.md` | 用户的纠正、偏好确认、工作方式反馈 | "用户希望回复简洁，不要加冗长总结" |
| **project** | `project.md` | 项目目标、技术决策、当前状态 | "项目使用 FastAPI，数据库用 PostgreSQL" |
| **reference** | `reference.md` | 外部系统指针（文档链接、配置位置等） | "API 文档在 https://docs.example.com/api" |

### 2.2 两级索引架构

这是记忆系统的核心设计，分为 **索引层**（`MEMORY.md`）和 **文件层**（独立 `.md` 文件）。

#### 索引层：MEMORY.md

```markdown
# NexusAgent Memory Index
<!-- AUTO-GENERATED: Do not edit manually. -->

## user
- [python_dev](memory/user.md) -- User is a Python backend engineer

## feedback
- [concise_replies](memory/feedback.md) -- User prefers short responses
```

**为什么需要索引？**
- 快速浏览：不用打开每个文件就能看到所有记忆的元数据（名称、类型、描述）
- 去重：`name + memory_type` 唯一键，避免重复记忆
- 分类管理：按类型分组，方便按类型查询/统计

#### 文件层：独立 .md 文件 + YAML frontmatter

```markdown
---
name: python_dev
type: user
description: User is a Python backend engineer
created: 2026-04-19T10:30:00
---

用户是 Python 后端工程师，主要使用 FastAPI 框架。
熟悉 Go，但不熟悉前端技术（React/TypeScript）。
```

**frontmatter 的作用**：提供机器可读的元数据（类型、描述、创建时间），同时人类也能直接阅读文件内容。

### 2.3 MemoryIndex 类（`memory/index.py`）

负责维护 `MEMORY.md` 的索引管理：

```python
class MemoryIndex:
    VALID_TYPES = ["user", "feedback", "project", "reference"]
    _ENTRY_RE = re.compile(r"^-\s+\[([^\]]+)\]\(([^)]+)\)\s*(?:--\s+(.*))?$")

    def add_entry(self, entry) -> bool:
        """添加条目，重复则更新而非新增"""
        existing = self._find_entry(entry.name, entry.memory_type)
        if existing:
            existing.file_path = entry.file_path
            existing.description = entry.description
            return False  # 更新
        self.entries.append(entry)
        return True  # 新增

    def remove_entry(self, name, memory_type) -> bool:
        """从索引中移除条目"""

    def list_entries(self, memory_type=None) -> list[MemoryEntry]:
        """列出所有条目，可按类型过滤"""
```

**关键细节——重复检测**：
`_find_entry(name, memory_type)` 使用 `name + memory_type` 组合作为唯一键。这意味着：
- 同一个名称可以在不同类型中各出现一次（`python_dev` 可以是 user 类型和 project 类型）
- 同一类型中同名的记忆会被更新而不是新增

### 2.4 MemorySystem 类（`memory/memory.py`）

这是面向上层的核心 API：

```python
class MemorySystem:
    def load_all(self) -> str:
        """加载所有记忆文件并合并为字符串"""

    def get(self, name: str) -> str:
        """获取单个记忆文件的内容"""

    def save(self, name, content, description) -> None:
        """保存记忆文件 + 更新索引"""

    def append(self, name, content) -> None:
        """追加到现有记忆文件"""

    def build_system_prompt_section(self) -> str:
        """格式化所有记忆，注入到 LLM 系统提示"""
```

#### load_all 的 frontmatter 处理

```python
def load_all(self) -> str:
    parts = []
    for name in self.MEMORY_TYPES:
        path = self.memory_dir / f"{name}.md"
        if path.exists():
            content = path.read_text(encoding="utf-8").strip()
            if content:
                meta, body = parse_frontmatter(content)  # 分离元数据和正文
                display = body if body else content       # 注入系统提示只需要正文
                parts.append(f"## {name.capitalize()} Memory\n{display}")
    return "\n\n".join(parts)
```

**关键设计**：注入系统提示时，**只使用正文部分**（frontmatter 后面的内容），不把元数据（name/type/description）喂给 LLM，节省 token。

## 三、记忆写入机制

### 3.1 MemoryWriteTool（`tools/builtin/memory_write.py`）

这是 LLM 和用户操作记忆的**唯一入口**，提供 4 种操作：

| 操作 | 用途 | 举例 |
|------|------|------|
| **save** | 创建新记忆或更新已有记忆 | LLM 发现用户偏好后保存 |
| **update** | 仅修改已有记忆 | 用户修改偏好时更新 |
| **forget** | 删除记忆 | 用户说"忘掉这个" |
| **list** | 列出所有记忆 | 用户问"你记得什么" |

#### 写入流程

```
LLM 调用: MemoryWrite(operation="save", name="python_dev", memory_type="user",
                      content="用户是 Python 工程师...", description="用户身份")
         │
         ▼
MemoryWriteTool._save()
    1. 验证: name 不为空, memory_type 在 VALID_TYPES 中
    2. 验证: content 长度 <= 10000 字符（防止超长记忆）
    3. 检查索引: name+type 是否已存在？
       - 存在 → index.update_entry() 更新描述
       - 不存在 → index.add_entry() 新增条目
    4. 写文件:
       meta = {name, type, description, created}
       format_frontmatter(meta, content) → "user.md"
    5. 保存索引: index.save() → 更新 MEMORY.md
```

#### 内容长度限制

```python
MAX_CONTENT_LENGTH = 10000
```

**为什么限制？** 防止一条超长记忆挤占系统提示的 token 预算。如果内容超过 10000 字符，LLM 会收到错误提示："Split into multiple shorter memories."

#### 忘记记忆的特殊处理

```python
def _forget(self, name, memory_type):
    # 从索引中移除
    self.index.remove_entry(name, memory_type)
    self.index.save()

    # 处理文件
    if file_name in STANDARD_FILES:  # user.md, feedback.md, project.md, reference.md
        file_path.write_text("")     # 清空内容，不删除文件
    else:
        file_path.unlink()           # 动态文件直接删除
```

**为什么标准文件要清空而不是删除？** 这四个文件是记忆系统的"骨架"，`MemorySystem.__init__` 会检查它们是否存在，不存在则创建。如果直接删除了，下次初始化又会重建一个空文件。清空内容既达到了"忘记"的效果，又保持了文件结构稳定。

### 3.2 自动记忆触发

记忆系统通过 **工具** 的方式暴露给 LLM，LLM 可以在对话过程中自主决定何时保存记忆。典型场景：

1. **用户纠正后**：用户说"别用英文回复"，LLM 调用 `MemoryWrite(save, "english_preference", "feedback", "用户偏好中文回复")`
2. **项目信息发现**：LLM 读到 `nexus.toml` 后调用 `MemoryWrite(save, "tech_stack", "project", "项目使用 Python 3.12 + FastAPI")`
3. **用户身份推断**：通过对话模式推断用户角色，自动保存

**权限配置**：`MemoryWrite` 的权限级别是 `"approve"`（自动批准，见 `permission/policy.py:28`），这意味着 LLM 可以无阻碍地保存记忆，不需要用户确认。

## 四、单轮会话（Session）

### 4.1 SessionManager（`memory/session.py`）

负责将每次对话的完整消息历史持久化到 JSON 文件：

```
.nexus/sessions/
├── 20260419_103000.json    ← 2026年4月19日 10:30:00 的会话
├── 20260419_141500.json    ← 同日 14:15:00 的会话
└── ...
```

#### JSON 格式

```json
{
  "id": "20260419_103000",
  "created_at": "2026-04-19T10:30:00",
  "messages": [
    {"role": "user", "content": "帮我修 auth.py 的 bug", "token_count": 12, "metadata": {}},
    {"role": "assistant", "content": "好的，让我先看看...", "token_count": 45, "metadata": {}},
    ...
  ]
}
```

#### 保存时机

在 `master.py` 的主循环中，Session 在两个时机被保存：

1. **每轮对话正常结束时**（`master.py:375`）：
   ```python
   # 自动保存会话
   self._save_session()
   ```

2. **用户取消时**（`master.py:147`）：
   ```python
   def _check_cancelled(self) -> bool:
       if self._cancel_event.is_set():
           self._save_checkpoint("cancelled")
           self._save_session()  # 确保中断状态也被保存
           return True
   ```

### 4.2 会话与记忆的区别

| | 会话（Session） | 记忆（Memory） |
|---|---|---|
| **内容** | 完整对话消息（User/Assistant/Tool） | 提取的关键信息（偏好、决策） |
| **用途** | 回顾历史、断点恢复 | 指导 Agent 行为 |
| **注入系统提示** | 否 | 是（通过 `build_system_prompt_section`） |
| **格式** | JSON | Markdown + YAML frontmatter |
| **写入者** | MasterAgent 自动保存 | LLM 调用 MemoryWriteTool |

## 五、记忆注入系统提示的完整链路

记忆从存储到影响 LLM 行为，需要经过以下链路：

### 5.1 启动时加载（`main.py:76-82`）

```python
# 创建记忆系统
memory = MemorySystem(workdir / ".nexus" / "memory")

# 加载所有记忆文件
memory_content = memory.load_all()

# 注入到 PromptBuilder 的 "memory" 片段
if memory_content:
    builder.update_section("memory", memory_content)
```

### 5.2 PromptBuilder 的结构化处理（`context/builder.py:41`）

```python
SECTION_ORDER = ["role", "rules", "project", "memory", "skills"]
```

记忆被放入 `memory` 片段，在系统提示中排在第 4 位：

```
## ROLE
你是 NexusAgent，一个运行在终端环境中的 AI 编程助手。

## RULES
工作方式：先读文件再修改...

## PROJECT
（来自 .nexus.md 的项目上下文）

## MEMORY
## User Memory
用户是 Python 后端工程师...

## Feedback Memory
用户偏好中文回复...

## SKILLS
（可用的技能列表）
```

### 5.3 Anthropic 缓存策略

`PromptSection.cacheable = True`（默认值），意味着记忆片段会被标记为 `cache_control: {"type": "ephemeral"}`，利用 Anthropic 的提示缓存机制，在多次对话轮次中复用缓存，降低 token 成本。

## 六、设计权衡

### 6.1 为什么用 Markdown 文件而不是数据库

- **可读性**：用户可以直接用编辑器打开 `.md` 文件查看/修改
- **Git 友好**：记忆文件可以纳入版本控制，追踪记忆变更历史
- **无外部依赖**：不需要安装数据库，Python 标准库的 `pathlib` + 正则即可
- **灵活性**：frontmatter + 正文的两层结构兼顾机器可读和人类可读

### 6.2 为什么记忆系统没有压缩/淘汰机制

配置中有 `max_memories_per_type: int = 50`（`config.py:46`），但当前代码中**未实际强制执行**。这意味着记忆会无限增长。

**潜在问题**：如果用户长期使用，`user.md` 等文件可能变得很大，挤占系统提示的 token 预算。

**未来可能的改进**：
1. 实现自动淘汰：超过 50 条时移除最早的条目
2. 记忆重要性评分：让 LLM 评估每条记忆的价值
3. 分级加载：只加载最相关的记忆，而不是全部

### 6.3 为什么 MemoryWrite 不在 TUI 中注册为 / 命令

当前 TUI 注册了 `/memory`、`/forget`、`/memstats` 等命令（`main.py:150-187`），但这些命令直接调用 `agent.memory_write_tool.execute()`，绕过了 LLM。这意味着：
- 用户可以通过命令直接管理记忆
- LLM 也可以通过工具调用来管理记忆
- 两条路径操作同一个 `MemoryIndex` 实例

**潜在竞态条件**：如果用户和 LLM 同时操作记忆，索引可能不一致。但实际中这种情况极少发生，因为用户输入和 LLM 工具调用是串行的。
