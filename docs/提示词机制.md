# 提示词与上下文注入机制

## 一、概述

NexusAgent 的提示词（Prompt）系统负责**将结构化系统提示、项目上下文、记忆、技能定义和对话历史组装成 LLM API 可接受的格式**。类比 Web 开发中的模板引擎：将静态模板（角色定义、规则）、动态数据（项目上下文、记忆）、和运行时状态（对话历史）合并为最终渲染结果。

### 为什么需要结构化提示词

如果直接把所有内容拼成一个长字符串：
- **无法缓存**：每次请求都重新计算全部 token，API 成本高
- **难以维护**：各模块（记忆、项目、技能）互相耦合，修改一处影响全局
- **提供商不兼容**：Anthropic 使用 content blocks + `cache_control`，OpenAI 使用纯文本 system 消息

结构化方案将系统提示拆为独立片段（section），每个片段可独立更新、缓存、注入。

### 整体架构


## 二、核心组件详解

### 2.1 PromptBuilder — 结构化提示组装器（`context/builder.py:1-80`）

```python
SECTION_ORDER = ["role", "rules", "project", "memory", "skills"]

class PromptSection:
    def __init__(self, name: str, content: str = "", cacheable: bool = True):
        self.name = name
        self.content = content
        self.cacheable = cacheable    # 是否标记 Anthropic 缓存

class PromptBuilder:
    def __init__(self):
        self.sections: dict[str, PromptSection] = {}
        for name in SECTION_ORDER:
            self.sections[name] = PromptSection(name)
        self._load_defaults()          # 从 skills/default_prompts.yaml 加载 role/rules

    def update_section(self, name: str, content: str) -> bool:
        """更新单个片段内容"""

    def get_section(self, name: str) -> str | None:
        """获取单个片段内容"""

    def build(self) -> dict:
        """构建两种格式的系统提示"""
        return {"blocks": self._build_blocks(), "text": self._build_text()}
```

**关键设计——默认片段加载**：
`_load_defaults()` 从 `skills/default_prompts.yaml` 读取 `role` 和 `rules` 两个静态片段的默认内容。这两个片段是"模板骨架"，不随项目变化。而 `project`、`memory`、`skills` 三个片段在运行时动态注入。

#### 两种输出格式

| 格式 | 用途 | 结构 |
|------|------|------|
| **blocks** | Anthropic API | `[{"type":"text","text":"...","cache_control":{"type":"ephemeral"}}, ...]` |
| **text** | OpenAI 兼容 API | `"## ROLE\n...\n\n## RULES\n...\n\n## PROJECT\n..."` |

**`cacheable` 标志的作用**：每个片段默认 `cacheable=True`，在 `_build_blocks()` 中会被标记为 `cache_control: {"type": "ephemeral"}`。Anthropic 会缓存最后一个带此标记的 block 之前的所有内容。这意味着：
- 角色定义和规则在多次对话中几乎不变 → 缓存命中率高 → 降低 latency 和成本
- 当只有 `memory` 片段变化时，`role`/`rules`/`project` 的缓存仍然有效

### 2.2 项目上下文加载（`context/project_context.py`）

```python
def load_project_context(project_root: Path) -> str:
    """加载项目根目录的 .nexus.md 文件"""

def load_hierarchy_context(start_dir: Path, project_root: Path) -> str:
    """从当前目录向上遍历，收集所有 .nexus.md 文件"""
```

**层级加载逻辑**（`project_context.py:30-55`）：

```python
def load_hierarchy_context(start_dir, project_root):
    parts = []
    current = start_dir
    while current != project_root.parent:
        nexus_file = current / ".nexus.md"
        if nexus_file.exists():
            parts.append(nexus_file.read_text())
        if current == project_root:
            break
        current = current.parent
    return "\n\n---\n\n".join(parts)
```

**关键设计——路径优先级**：从当前工作目录开始向上遍历，离工作目录越近的文件**越晚追加**到结果中。这意味着：
- `src/frontend/.nexus.md`（近）的内容排在最后
- `project_root/.nexus.md`（远）的内容排在前面
- LLM 处理时，后面的内容在系统提示中更"新鲜"，权重更高

### 2.3 ContextManager — 上下文生命周期管理（`context/manager.py:12-237`）

`ContextManager` 是 PromptBuilder 的上层协调者，负责：

```python
            return result["blocks"]    # 带 cache_control 的 content blocks
        return result["text"]          # 纯文本字符串

    def build_messages(self) -> list[dict]:
        """构建对话历史消息列表"""
        # Anthropic: 直接 to_api_format()
        # OpenAI: ToolResultMessage 转为 {"role": "tool", "tool_call_id": ..., "content": ...}
```

**提供商适配**：ToolResultMessage 在 OpenAI 格式下需要特殊处理——Anthropic 将工具结果作为 `user` 角色的 content block（`type: "tool_result"`），而 OpenAI 需要独立的 `{"role": "tool"}` 消息。

### 2.4 AnthropicClient — 系统提示块转换（`llm/anthropic.py`）

```python
def _build_system_prompt(self, system: str | list[dict]) -> list[dict]:
    if isinstance(system, str):
        return [{"type": "text", "text": system}]
    return system    # 已经是 content blocks 列表（来自 PromptBuilder）
```

**关键设计——extended thinking 模式**：
```python
if self.thinking_budget > 0:
    temperature = 1.0    # Anthropic 要求 thinking 模式必须 temperature=1.0
```

当 `thinking_budget > 0` 时，API 请求中会添加 `thinking: {"type": "enabled", "budget_tokens": ...}`。此时流式响应会包含 `thinking_delta` 事件，需要在处理时跳过（不展示给用户）。

### 2.5 OpenAICompatibleClient — 系统提示作为首条消息（`llm/openai_compat.py`）

```python
# 系统提示作为第一条 system 角色消息
api_messages = [{"role": "system", "content": system_prompt}, *messages]

# 工具定义包装为 OpenAI 格式
api_tools = [{"type": "function", "function": t} for t in tools]
```

**与 Anthropic 的关键差异**：
- Anthropic 的 system prompt 是独立参数（`system=`），OpenAI 的 system prompt 是消息列表的第一条
- OpenAI 的工具定义需要 `{"type": "function", "function": {...}}` 包装
- 流式响应中，工具调用参数可能分布在多个 chunk 中，需要累积拼接

## 四、关键设计决策

### 4.1 为什么用 content blocks 而不是纯文本（Anthropic）

Anthropic API 支持两种系统提示格式：
- **纯文本字符串**：简单但无法控制缓存粒度
- **content blocks 列表**：每个 block 可独立设置 `cache_control`

选择 content blocks 的原因：
- **缓存粒度**：`role`、`rules`、`project` 片段在多次对话中不变，可以被缓存复用
- **成本优化**：Anthropic 的缓存读取价格远低于重新计算（约 1/10）
- **灵活性**：未来可以为不同片段设置不同的缓存策略

### 4.2 为什么 PromptBuilder 的片段顺序固定

`SECTION_ORDER = ["role", "rules", "project", "memory", "skills"]` 的顺序经过设计：

| 顺序 | 片段 | 原因 |
|------|------|------|
| 1 | role | 身份定义，LLM 最先建立自我认知 |
| 2 | rules | 行为约束，紧跟身份 |
| 3 | project | 项目背景，为后续记忆提供上下文 |
| 4 | memory | 用户偏好和经验教训，在中间位置 |
| 5 | skills | 技能指令，在最末尾（LLM 对末尾内容关注度更高） |

**为什么 skills 在最后？** LLM 对系统提示的开头和结尾关注度最高（"U 型注意力"）。技能指令需要 LLM 在识别到用户触发时严格执行，放在末尾可以提高遵从度。

### 4.3 Anthropic 缓存策略的细节

```python
# PromptBuilder._build_blocks() 的逻辑
for section in sections:
    block = {"type": "text", "text": section.content}
    if section.cacheable:
        block["cache_control"] = {"type": "ephemeral"}
    blocks.append(block)
```

**Anthropic 缓存机制**：
- 只有**最后一个**带 `cache_control` 的 block 会被实际标记为缓存点
- 之前的 `cache_control` 标记会被忽略
- 当前所有片段都标记为 `cacheable=True`，意味着最后一个片段（skills）是缓存点
- 当 skills 内容不变时，整个系统提示都被缓存；当 skills 变化时，缓存失效

**潜在优化空间**：可以为不同片段设置不同的 `cacheable` 值。例如，`role` 和 `rules` 几乎不变，可以设为 `True`；`memory` 和 `skills` 可能动态变化，设为 `False`。这样缓存点始终落在最后一个静态片段上，获得更稳定的缓存命中率。

### 4.4 Extended Thinking 模式的约束

Anthropic 的 extended thinking 模式有两个硬性约束：
1. **temperature 必须为 1.0**（`llm/anthropic.py:63-65`）
2. **thinking budget 不能超过总 max_tokens**

```python
if self.thinking_budget > 0:
    temperature = 1.0  # Anthropic 要求 thinking 模式 temperature=1.0
```

**为什么强制 temperature=1.0？** Anthropic 的 thinking 模式内部使用特定采样策略，与低 temperature 不兼容。如果用户配置了 `thinking_budget > 0` 但 `temperature != 1.0`，API 会返回错误。

### 4.5 系统提示在压缩中的保护

上下文压缩（compaction）只针对**对话历史消息**，不影响系统提示。系统提示由 PromptBuilder 在每次 LLM 调用前重新构建（`master.py:215`），这意味着：
- 如果用户在对话过程中修改了记忆，下一轮 LLM 调用会自动看到更新后的记忆
- 项目上下文文件如果在运行中被修改，**不会**自动重新加载（只在启动时加载一次）

### 4.6 已知问题

#### 4.6.1 项目上下文不热重载

`.nexus.md` 文件只在 `main.py` 启动时加载一次。如果用户在 Agent 运行期间修改了 `.nexus.md`，新内容不会被注入到系统提示中。需要重启 Agent 或手动 `/compact` 触发重建。

#### 4.6.2 技能内容注入格式不统一

技能注入 system prompt 时用的是英文模板字符串（`main.py:104-107`），而 PromptBuilder 的其他片段用中文。如果用户配置了中文系统提示，技能描述突然出现英文会打断一致性。

#### 4.6.3 PromptBuilder 片段更新无验证

`update_section()` 不检查内容是否为空或过长。如果 `memory.load_all()` 返回超长内容（记忆文件无限增长，详见[记忆系统策略](./记忆系统策略.md) 6.2 节），系统提示可能超出 LLM 的 token 限制。
