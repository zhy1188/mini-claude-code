# 钩子系统：生命周期扩展机制

## 一、概述

钩子（Hook）系统允许用户**在 Agent 生命周期的关键节点注入自定义 Shell 命令**，无需修改代码即可扩展或拦截 Agent 行为。类比 Git 钩子（`pre-commit`、`post-merge`）或 Express 中间件。

### 为什么需要钩子

没有钩子时，如果要实现以下需求必须改代码：
- **审计**：记录每次工具调用的输入输出到日志文件
- **拦截**：禁止 LLM 执行某些特定模式的命令（如 `rm *.log`）
- **增强**：工具执行前自动注入环境变量或检查状态
- **通知**：工具执行完成后发送 Slack/邮件通知

钩子的核心价值：**扩展行为而不侵入核心循环**。

### 钩子与权限系统的区别

| 维度 | 权限系统（PermissionGate） | 钩子（Hook） |
|------|--------------------------|-------------|
| **目的** | 控制"能不能做" | 控制"做之前/之后还要做什么" |
| **交互方式** | 弹出确认框让用户选择 | 执行配置的 Shell 命令 |
| **结果** | 批准/拒绝 | 可以阻塞（命令失败）或不阻塞 |
| **类比** | 门禁卡 | 监控摄像头 + 自动门 |

## 二、核心组件

### 2.1 六种生命周期钩子（`hooks/types.py:10-18`）

```python
class HookType(Enum):
    PRE_USER_MESSAGE = "pre_user_message"    # 处理用户输入之前
    POST_USER_MESSAGE = "post_user_message"  # 处理用户输入之后
    PRE_TOOL_USE = "pre_tool_use"           # 执行工具之前
    POST_TOOL_USE = "post_tool_use"         # 执行工具之后
    PRE_RESPONSE = "pre_response"           # 发送 LLM 响应之前
    POST_RESPONSE = "post_response"         # 发送 LLM 响应之后
```

**每种钩子的触发时机和可用上下文变量**：

| 钩子类型 | 触发时机 | 上下文变量 | 典型用途 |
|---------|---------|-----------|---------|
| `PRE_USER_MESSAGE` | 收到用户输入后、处理之前 | `user_input` | 输入过滤、敏感词检测 |
| `POST_USER_MESSAGE` | 整轮对话完成后 | `user_input`, `state` | 会话结束后的清理工作 |
| `PRE_TOOL_USE` | 工具执行前、权限检查后 | `tool_name`, `tool_input` | 审计日志、动态拦截 |
| `POST_TOOL_USE` | 工具执行完成后 | `tool_name`, `tool_input`, `tool_output`, `status` | 结果验证、通知 |
| `PRE_RESPONSE` | LLM 回复前（当前未接入） | `response` | 回复内容审核 |
| `POST_RESPONSE` | LLM 回复发送后 | `response` | 回复内容归档 |

### 2.2 HookConfig — 钩子配置（`hooks/types.py:21-40`）

```python
class HookConfig(BaseModel):
    hook_type: HookType        # 在哪个生命周期触发
    matcher: str               # 工具名称或通配符 "*"
    command: str               # 要执行的 Shell 命令
    blocking: bool = False     # 是否阻塞 Agent（失败时阻止继续）
```

**匹配逻辑**（`hooks/types.py:35-40`）：

```python
def matches(self, context: dict) -> bool:
    if self.matcher == "*":
        return True                    # 通配符：匹配所有工具
    tool_name = context.get("tool_name", "")
    return self.matcher == tool_name   # 精确匹配工具名称
```

**关键设计**：`matcher` 目前只支持**精确匹配**和**全局通配符 `*`**，不支持正则表达式或 glob 模式。这意味着无法写 `"Bash*"` 或 `"Write|Read"` 这样的模式。

### 2.3 HookResult — 钩子执行结果（`hooks/engine.py:12-16`）

```python
@dataclass
class HookResult:
    blocked: bool = False    # 是否阻塞后续操作
    reason: str = ""         # 阻塞原因
    output: str = ""         # 钩子命令的输出
```

## 三、HookEngine 执行引擎

### 3.1 注册与触发（`hooks/engine.py:45-69`）

```python
class HookEngine:
    def __init__(self):
        self.hooks: dict[HookType, list[HookConfig]] = {}

    def register(self, hook: HookConfig) -> None:
        """注册一个钩子"""
        if hook.hook_type not in self.hooks:
            self.hooks[hook.hook_type] = []
        self.hooks[hook.hook_type].append(hook)

    async def trigger(self, hook_type: HookType, context: dict) -> HookResult:
        """触发指定类型的所有钩子"""
        hooks = self.hooks.get(hook_type, [])
        for hook in hooks:
            if hook.matches(context):
                result = await self._execute_hook(hook, context)
                if result.blocked:
                    return result    # 短路：一个阻塞钩失败就停止
        return HookResult()          # 全部通过
```

**关键行为——短路机制**：同一类型的多个钩子按注册顺序执行，如果某个阻塞钩返回 `blocked=True`，**后续钩子不再执行**，直接返回失败结果。

### 3.2 钩子命令执行（`hooks/engine.py:71-103`）

```python
async def _execute_hook(self, hook: HookConfig, context: dict) -> HookResult:
    # 1. 变量插值
    cmd = self._interpolate(hook.command, context)

    # 2. 启动子进程
    proc = await asyncio.create_subprocess_shell(cmd, ...)

    # 3. 阻塞模式：等待结果，检查退出码
    if hook.blocking:
        stdout, stderr = await asyncio.wait_for(proc.communicate(), timeout=30)
        output = stdout.decode() + stderr.decode()
        if proc.returncode != 0:
            return HookResult(blocked=True, reason=f"Hook failed (exit {proc.returncode})")
        return HookResult(output=output)

    # 4. 非阻塞模式：后台执行，不等待
    else:
        asyncio.create_task(proc.communicate())
        return HookResult()
```

**阻塞 vs 非阻塞的对比**：

| 模式 | 执行方式 | 超时 | Agent 行为 | 失败影响 |
|------|---------|------|-----------|---------|
| **阻塞** | `await asyncio.wait_for(..., timeout=30)` | 30 秒 | 等待钩子完成 | 返回码非 0 → 阻塞后续操作 |
| **非阻塞** | `asyncio.create_task(...)` | 无 | 立即继续，不等待 | 即使失败也不影响 Agent |

### 3.3 变量插值（`hooks/engine.py:105-110`）

```python
def _interpolate(self, template: str, context: dict) -> str:
    return string.Template(template).safe_substitute(context)
```

使用 Python 标准库的 `string.Template`，语法为 `${variable}` 或 `$variable`：

```bash
# 钩子命令模板
echo '$tool_name called with $tool_input' >> audit.log

# 插值后
echo 'Bash called with {"command": "ls -la"}' >> audit.log
```

**`safe_substitute` 的特性**：如果上下文缺少某个变量，不会报错，而是保留原始 `$variable` 文本。这避免了因变量缺失导致钩子崩溃。

## 四、钩子在主循环中的集成点

钩子在 `master.py` 的 **5 个位置**被触发：

### 4.1 PRE_USER_MESSAGE（`master.py:165-173`）

```python
# 触发用户消息前钩子
if self.hook_engine:
    hook_result = await self.hook_engine.trigger(
        HookType.PRE_USER_MESSAGE, {"user_input": user_input}
    )
    if hook_result.blocked:
        await self.tui.show_status(f"Hook blocked: {hook_result.reason}")
        self.state_machine.transition("done")
        self._sync_legacy_state()
        return    # 直接结束本轮处理
```

**效果**：如果钩子阻塞，Agent 直接转换到 `done` 状态，不执行任何后续操作。

### 4.2 PRE_TOOL_USE（`master.py:280-292`）

```python
# 触发工具使用前钩子
if self.hook_engine:
    hook_ctx = {
        "tool_name": tc.name,
        "tool_input": json.dumps(tc.input),
    }
    hook_result = await self.hook_engine.trigger(
        HookType.PRE_TOOL_USE, hook_ctx
    )
    if hook_result.blocked:
        await self.tui.show_status(f"Hook blocked {tc.name}: {hook_result.reason}")
        self.tool_tracker.fail(tc.id, "Hook blocked")
        continue    # 跳过这个工具，继续处理下一个
```

**效果**：钩子阻塞时，标记该工具为 `failed`，跳过执行，但**不影响同批次的其他工具**。

### 4.3 POST_TOOL_USE（`master.py:312-321`）

```python
# 触发工具使用后钩子
if self.hook_engine:
    await self.hook_engine.trigger(
        HookType.POST_TOOL_USE,
        {
            "tool_name": tc.name,
            "tool_input": json.dumps(tc.input),
            "tool_output": result_content,
            "status": status,
        },
    )
```

**效果**：后置钩子通常是**非阻塞**的，用于审计、通知等。即使阻塞失败也不会影响工具结果（工具已执行完毕）。

### 4.4 POST_RESPONSE（`master.py:347-351`）

```python
# 触发响应后钩子
if self.hook_engine:
    await self.hook_engine.trigger(
        HookType.POST_RESPONSE,
        {"response": full_content},
    )
```

**效果**：LLM 返回文本回复后触发，可用于归档回复内容。

### 4.5 POST_USER_MESSAGE（`master.py:368-372`）

```python
# 触发用户消息后钩子
if self.hook_engine:
    await self.hook_engine.trigger(
        HookType.POST_USER_MESSAGE,
        {"user_input": user_input, "state": self.state.value},
    )
```

**效果**：整轮对话完成后触发，可用于会话统计、清理等。

## 五、典型使用场景

### 5.1 工具调用审计日志

```python
HookConfig(
    hook_type=HookType.PRE_TOOL_USE,
    matcher="*",
    command="echo '[${tool_name}] ${tool_input}' >> /tmp/nexus_audit.log",
    blocking=False,
)
```

### 5.2 危险命令二次检测

```python
HookConfig(
    hook_type=HookType.PRE_TOOL_USE,
    matcher="Bash",
    command="echo '${tool_input}' | grep -q 'rm -rf' && exit 1 || exit 0",
    blocking=True,
)
```

如果工具输入中包含 `rm -rf`，`grep` 匹配成功返回 0，但 `&& exit 1` 会让钩子返回非零退出码，从而阻塞工具执行。

### 5.3 工具输出通知

```python
HookConfig(
    hook_type=HookType.POST_TOOL_USE,
    matcher="Bash",
    command="curl -X POST https://hooks.slack.com/... -d '${tool_output:0:200}'",
    blocking=False,
)
```

### 5.4 输入过滤

```python
HookConfig(
    hook_type=HookType.PRE_USER_MESSAGE,
    matcher="*",
    command="echo '${user_input}' | grep -qi 'password\\|secret\\|token' && exit 1 || exit 0",
    blocking=True,
)
```

## 六、已知问题与不足

### 6.1 钩子未在启动时加载

检查 `main.py` 的初始化流程，**HookEngine 从未被创建或注册到 MasterAgent**。虽然 `MasterAgent.__init__` 接受 `hook_engine` 参数，`main.py` 中的 `_run_async()` 没有传入它：

```python
# main.py:245-253
agent = MasterAgent(
    llm_client=llm,
    tool_registry=registry,
    context_manager=context_mgr,
    permission_gate=permission_gate,
    tui=tui,
    workdir=workdir,
    # ← 没有 hook_engine 参数
    context_retriever=context_retriever,
)
```

同时 `config.py` 中也没有 `hooks` 配置段。这意味着**钩子系统当前是空壳**，代码完整但未被启用。

### 6.2 PRE_RESPONSE 和 POST_RESPONSE 的上下文不完整

`POST_RESPONSE` 只传了 `{"response": full_content}`，缺少 `tool_calls` 信息。如果 LLM 同时返回了文本和工具调用，钩子看不到工具调用部分。

### 6.3 matcher 只支持精确匹配

`HookConfig.matches()` 只支持 `==` 精确匹配和 `*` 全局通配符，不支持：
- 正则表达式（如 `"^(Bash|Write)$"`）
- glob 模式（如 `"*Tool"`）
- 列表匹配（如 `["Bash", "Write"]`）

### 6.4 阻塞超时硬编码为 30 秒

```python
stdout, stderr = await asyncio.wait_for(proc.communicate(), timeout=30)
```

30 秒超时是硬编码的，无法通过配置调整。对于需要网络请求的钩子（如调用外部 API）可能不够。

### 6.5 变量插值的安全隐患

`string.Template.safe_substitute()` 直接将上下文变量注入 Shell 命令，如果 `tool_input` 包含恶意 Shell 元字符（如 `; rm -rf /`），可能被注入执行。虽然钩子命令是管理员配置的（不是 LLM 控制的），但如果上下文变量来自 LLM 的工具调用输入，仍然存在间接注入风险。
