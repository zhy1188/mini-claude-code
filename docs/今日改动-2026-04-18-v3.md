# 今日改动 - 2026-04-18 (v3): 状态管理系统

## 一、有限状态机（FSM）

**问题**：`MasterAgent` 使用 `AgentState` 枚举直接赋值状态，无约束校验。任意状态可跳到任意状态，无法保证流程合法性。

**设计**：定义 `VALID_TRANSITIONS` 转换表，每次 `transition()` 调用前校验合法性，非法抛出 `InvalidStateTransition`。提供 `force()` 用于异常恢复。

**关键代码**：
- `agent/state_machine.py` — `StateMachine` dataclass，`transition()` 方法校验当前状态到目标状态的合法性
- `master.py` — 所有 `self.state = AgentState.X` 替换为 `self.state_machine.transition("x")`，用 try/except + `force()` 兜底

**面试话术**：
> "我用有限状态机替代了裸枚举。核心思路是用一张转换表定义所有合法路径，比如 idle 只能到 gathering，thinking 只能到 acting/verifying/done/error。每次状态变更都先查表校验，非法直接抛异常。同时也留了 force() 方法用于异常恢复，不至于死锁。这样整个 Agent 的生命周期从 4 个状态随便跳变成了严格的状态图。"

## 二、Checkpoint 检查点系统

**问题**：中断或崩溃时，对话中间状态（消息历史、工具执行进度）全部丢失，无法恢复。

**设计**：在三个关键节点写 checkpoint 到 `.nexus/checkpoints/`：
1. 用户输入添加后（保存 user_input）
2. LLM 调用前（保存 iteration 计数）
3. 工具执行完成后（保存 tool_executions 列表）

每个 checkpoint 是 JSON 文件，包含 state、messages、session_id、iteration 等。启动时 `resume()` 可加载最新 checkpoint 恢复对话。自动清理旧 checkpoint（保留最新 10 个）。

**关键代码**：
- `agent/checkpoint.py` — `Checkpoint.save()` / `load_latest()` / `has_pending()` / `_cleanup_old()`
- `master.py` — `run()` 中 3 处 `_save_checkpoint()` 调用；`resume()` 方法从 JSON 重建消息列表

**面试话术**：
> "借鉴了游戏存档的思路——在关键操作前自动存档。checkpoint 保存的是完整的对话历史和工具执行状态。如果进程崩溃或被 Ctrl+C 中断，下次启动时 `resume()` 会从 JSON 重建消息列表和 token 计数，接着上次中断的地方继续。同时做了自动清理，只保留最新 10 个，避免磁盘无限增长。"

## 三、异步中断（Ctrl+C 安全停止）

**问题**：无法安全打断正在执行的操作。Ctrl+C 会直接终止进程，中间状态丢失。

**设计**：用 `asyncio.Event` 作为取消信号。TUI 的 `Ctrl+C` 或 `/stop` 命令调用 `request_cancel()` 设置事件。在 5 个检查点轮询：
1. 主循环入口处
2. LLM 流式响应每个 token 后
3. 每个工具执行前

取消时保存 checkpoint 和 session，优雅退出到 done 状态。

**关键代码**：
- `master.py` — `_cancel_event = asyncio.Event()`，`request_cancel()` 设置，`_check_cancelled()` 检查并保存状态
- 流式响应中 `if self._cancel_event.is_set(): await self._graceful_stop()`

**面试话术**：
> "用 asyncio.Event 做了一个轻量的取消信号，比信号量或线程中断安全得多。在循环边界、流式 token 接收、工具执行前这三个关键位置都检查取消标志。取消不是立即终止，而是先保存 checkpoint 和 session，再优雅退出到 done 状态。这样用户按 Ctrl+C 后不会丢失上下文，下次还能恢复对话。"

## 四、工具执行状态追踪

**问题**：工具执行是黑盒，不知道哪些工具在跑、进度如何、结果是什么。中断时无法区分已完成和未完成的工具。

**设计**：`ToolExecution` 跟踪每个工具的生命周期：pending → running → done/failed/cancelled。记录 started_at/finished_at 计算执行时长。`ToolTracker` 提供 summary() 用于 checkpoint 序列化。

**关键代码**：
- `agent/tool_tracker.py` — `ToolExecution` dataclass（含 duration 属性），`ToolTracker` 管理所有执行
- `master.py` — 工具执行前 `tool_tracker.create()/start()`，执行后 `complete()/fail()`

**面试话术**：
> "每个工具调用都经过完整的生命周期管理：创建时是 pending，开始执行变成 running，成功是 done，失败是 failed，取消是 cancelled。用 time.monotonic() 记录精确时长。checkpoint 序列化时只保留 output 前 500 字符，避免大输出撑大存档。多个工具并行执行时，取消只会影响 pending/running 的，已完成的保持结果。这对面试演示时可以直观看到工具执行的完整状态图。"
