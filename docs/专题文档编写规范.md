# 专题文档编写规范

## 一、文档定位

专题文档用于**深入解释 NexusAgent 中某个模块或机制的完整逻辑**。面向的读者是需要理解代码 "为什么这样设计、是什么、怎么实现" 的开发者。

---

## 二、核心结构

所有专题文档遵循 **总 → 分** 的层级结构：

```
一、概述          ← 全局视角：是什么、为什么、整体架构
二、核心组件详解   ← 逐个模块拆解
三、完整流程/链路  ← 端到端的执行路径
四、关键设计决策   ← 权衡、取舍、已知问题
五、涉及文件       ← 文件清单速查
```

### 每一节内部：三问驱动

每个概念/组件的解释都回答三个问题：

| 问题 | 对应内容 | 目的 |
|------|---------|------|
| **为什么** | 设计动机、要解决的问题、替代方案的不足 | 帮助读者理解设计意图 |
| **是什么** | 概念定义、数据结构、类的职责 | 建立正确的心智模型 |
| **怎么做** | 代码路径、执行流程、关键逻辑 | 能够定位和修改代码 |

---

## 三、格式规范

### 3.1 标题层级

```markdown
# 专题文档标题           ← 文件名对应的中文标题（一级标题）

## 一、概述               ← 大节（二级标题，用中文数字编号）

### 3.1 小节标题          ← 小节（三级标题，用数字编号）

#### 3.1.1 子小节         ← 需要时用四级标题
```

**规则**：
- 一级标题：与文件名一致，描述模块/机制名称
- 二级标题：用 "一、二、三、四、五" 中文数字
- 三级及以下：用阿拉伯数字编号

### 3.2 代码引用

引用代码时**必须标注文件路径和关键行号范围**：

```markdown
定义在 `tools/base.py:48-65`：
```python
def _resolve_path(self, path: str) -> Path:
    ...
```
```

**规则**：
- 路径用反引号包裹：`tools/base.py:48-65`
- 只展示关键行，不贴大段代码
- 代码块中的注释用中文

### 3.3 表格使用

用于并列比较、分类说明：

```markdown
| 类型 | 系统提示 | 可用工具 | 用途 |
|------|---------|---------|------|
| general | "你是..." | Read, Write, Bash | 通用任务 |
| research | "你是..." | Read, Glob, Grep | 只读分析 |
```

**规则**：
- 列头要简短（1-4 个词）
- 内容要具体，不写空话

### 3.4 架构图/流程图

使用 ASCII art 绘制架构和流程：

```
┌─────────────────┐
│   MasterAgent   │
│   ┌───────────┐ │
│   │ Context   │ │
│   └───────────┘ │
└────────┬────────┘
         ▼
   ┌───────────┐
   │ SubAgent  │
   └───────────┘
```

**规则**：
- 用 `─` `│` `┌` `┐` `└` `┘` `▼` `▲` `←` `→` 等标准字符
- 框内文字居中对齐（不严格要求，但要可读）
- 复杂流程用编号步骤 + 箭头

### 3.5 代码块中的注释

所有代码块中的注释**必须用中文**：

```python
# 正确
# 从索引中移除条目
self.index.remove_entry(name, memory_type)

# 错误
# Remove entry from index
self.index.remove_entry(name, memory_type)
```

### 3.6 关键设计点的高亮

用加粗或引用块突出设计决策：

```markdown
**关键设计**：SubAgent 创建了一个**全新的 `ToolRegistry` 实例**，
只包含被分配的工具。它不持有主注册表的引用。
```

```markdown
> **为什么限制？** 防止一条超长记忆挤占系统提示的 token 预算。
```

## 四、内容规范

### 4.1 概述节必须包含

1. **一句话定义**：这个模块/机制是什么
2. **设计动机**：为什么需要它，解决了什么问题
3. **架构总览图**：用 ASCII art 或文字描述整体结构
4. **核心组件列表**：本节将详细讲解的组件

### 4.2 核心组件节必须包含

对每个组件：
1. **定义位置**：文件路径 + 行号
2. **核心代码**：展示最关键的类/函数定义
3. **关键特性**：用列表或表格列出设计要点
4. **与其他组件的关系**：如何被调用、调用谁

### 4.3 流程/链路节必须包含

1. **起点和终点**：从哪里开始，到哪里结束
2. **中间步骤**：按顺序编号，每步说明做了什么
3. **状态变化**：关键节点的状态转换（如果有）
4. **数据变化**：输入是什么，输出是什么

### 4.4 设计权衡节必须包含

1. **为什么选 A 不选 B**：至少说明一个替代方案及不选的理由
2. **已知问题/Bug**：如实标注发现的 bug 或不足
3. **未来改进方向**：可能的优化点

### 4.5 涉及文件节

用表格列出所有相关文件和各自职责：

```markdown
| 文件 | 职责 |
|------|------|
| `agent/state_machine.py` | FSM 状态机，转换表，验证逻辑 |
| `agent/master.py` | 主循环中状态转换的执行和兜底 |
```

**规则**：
- 文件路径用相对路径（相对于 `src/nexusagent/` 或项目根）
- 职责描述不超过 20 字
- 按重要性排序，核心文件在前

## 五、写作风格

### 5.1 语言

- 全部使用**中文**
- 技术术语可以保留英文（如 FSM、token、schema），但首次出现时需附带中文解释
- 避免口语化，保持技术文档的严谨性

### 5.2 详略程度

- **详写**：设计决策的原因、关键逻辑的细节、容易被误解的地方
- **略写**：样板代码、显而易见的逻辑（如 getter/setter）、与主题无关的细节

### 5.3 禁忌

- ❌ 不要写"显而易见"、"众所周知"这类话
- ❌ 不要假设读者已经了解模块的背景，每个专题应自包含
- ❌ 不要只贴代码不解释，代码是论据，解释才是论点
- ❌ 不要写多段落的文档字符串，一行注释说明为什么即可
- ❌ 不要使用 emoji 表情

### 5.4 好的 vs 不好的

**不好的**：
> ReadTool 用于读取文件内容。

**好的**：
> ReadTool 负责读取工作目录内的文件。关键特性是支持 `start_line` / `end_line` 行范围参数，避免 LLM 一次性读取几十 MB 的大文件。输出带行号（`f"{i:6d}  {line}"`），方便 LLM 在 `str_replace` 时准确定位要修改的代码行。

---

## 六、文件命名

- 使用**中文文件名**，描述模块/机制名称
- 用 `.md` 扩展名
- 示例：
  - `状态流转机制.md`
  - `主Agent与子Agent协作机制.md`
  - `记忆系统策略.md`
  - `工具系统全链路.md`

---

## 七、文档模板

新建专题文档时，可以套用以下骨架：

```markdown
# [模块名称]

## 一、概述

[一句话定义]

### 为什么需要[模块]

[设计动机、替代方案的不足]

### 整体架构

[ASCII 架构图或结构描述]

## 二、核心组件详解

### 2.1 [组件名称]（`文件路径:行号`）

[组件定义 + 关键代码 + 设计要点]

### 2.2 [组件名称]（`文件路径:行号`）

[同上]

## 三、完整流程/链路

[从起点到终点的编号步骤 + ASCII 流程图]

## 四、关键设计决策

### 4.1 [决策点 1]

[为什么这样设计、替代方案]

### 4.2 [决策点 2]

[同上]

### 4.3 已知问题

[如实标注 bug 或不足]

## 五、涉及文件

| 文件 | 职责 |
|------|------|
| ... | ... |
```

---

## 八、交叉引用

如果专题文档涉及其他专题中已讲解的内容，用 Markdown 链接引用：

```markdown
详见[主Agent与子Agent协作机制](./主Agent与子Agent协作机制.md)。
```

避免重复解释，保持文档之间的关联性。
