# 工具系统：从定义到执行的全链路

## 一、概述

NexusAgent 的工具系统实现了 **声明式定义 → JSON Schema 自动生成 → LLM 调用 → 本地执行** 的完整闭环。每个工具是一个 Python 类，通过继承 `Tool` 基类，只需定义 `name`、`description`、`parameters` 三个类属性和一个 `execute()` 方法，就能被 LLM 发现、理解并调用。

### 为什么这样设计

传统 AI 工具的常见做法是手动编写 JSON Schema 描述工具参数，然后写执行逻辑。这种方式的问题：
- **Schema 和逻辑分离**：改参数要同时改 JSON 和代码，容易不同步
- **扩展困难**：加一个新工具需要写大量样板代码
- **多提供商适配**：不同 LLM API 的 tool 格式略有差异，需要分别处理

NexusAgent 的方案是：**用 Python 类属性声明，运行时自动转译为 LLM API 需要的 JSON Schema**。改一个地方，所有格式自动更新。

## 三、核心组件详解

### 3.1 Tool 基类（`tools/base.py`）

所有工具的抽象基类，定义了三个层次：

#### 3.1.1 声明层：类属性

```python
class Tool(ABC):
    name: str = ""            # 工具名称（LLM 看到的名称）
    description: str = ""     # 工具描述（告诉 LLM 这个工具做什么）
    parameters: dict = {}     # 参数定义（JSON Schema 格式）
```

每个子类只需覆盖这三个属性，就能定义自己的接口。以 ReadTool 为例（`tools/builtin/read.py`）：

```python
class ReadTool(Tool):
    name = "Read"
    description = "Read the contents of a file. Supports line range limits."
    parameters = {
        "file_path": {
            "type": "string",
            "description": "Path to the file to read",
            "required": True,
        },
        "start_line": {
            "type": "integer",
            "description": "Start line number (1-indexed, default: 1)",
            "required": False,
        },
        "end_line": {
            "type": "integer",
            "description": "End line number (inclusive, default: last line)",
            "required": False,
        },
    }
```

**关键设计**：`parameters` 使用 Python dict 而非 JSON 字符串，好处是：
- IDE 自动补全、类型检查
- 支持注释（JSON 不支持注释）
- 可以在代码中动态添加参数

#### 3.1.2 转换层：to_llm_schema()

```python
def to_llm_schema(self) -> dict:
    """转换为 LLM API JSON Schema 格式"""
    # 1. 提取 required 字段名列表
    required = [
        name for name, spec in self.parameters.items()
        if spec.get("required", False)
    ]
    # 2. 构建 properties（去掉内部的 "required" 标记）
    properties = {}
    for name, spec in self.parameters.items():
        prop = {k: v for k, v in spec.items() if k != "required"}
        properties[name] = prop

    # 3. 组装为标准格式
    return {
        "name": self.name,
        "description": self.description,
        "input_schema": {
            "type": "object",
            "properties": properties,
            "required": required,
        },
    }
```

**转换逻辑解读**：
- `parameters` 中的每个参数定义包含 `type`、`description`、`required` 等字段
- `required` 在 `parameters` 中是**每个参数内部的布尔值**（如 `"required": True`）
- 但 LLM API 要求 `required` 是**顶层的字符串数组**（如 `"required": ["file_path"]`）
- `to_llm_schema()` 负责完成这个转换

生成的标准输出（以 ReadTool 为例）：
```json
{
  "name": "Read",
  "description": "Read the contents of a file.",
  "input_schema": {
    "type": "object",
    "properties": {
      "file_path": { "type": "string", "description": "Path to the file" },
      "start_line": { "type": "integer", "description": "Start line number" },
      "end_line": { "type": "integer", "description": "End line number" }
    },
    "required": ["file_path"]
  }
}
```

#### 3.1.3 安全层：沙箱和安全读取

```python
def _resolve_path(self, path: str) -> Path:
    """解析相对于 workdir 的路径，执行沙箱限制"""
    p = Path(path)
    workdir = Path(self.workdir)
    if p.is_absolute():
        resolved = p.resolve()
    else:
        resolved = (workdir / p).resolve()

    # 沙箱：确保路径在 workdir 范围内
    workdir_resolved = workdir.resolve()
    try:
        resolved.relative_to(workdir_resolved)
    except ValueError:
        raise PermissionError(
            f"Path '{path}' escapes workdir sandbox ({workdir_resolved})"
        )
    return resolved

def _safe_read(self, path: Path, max_bytes: int = 50_000) -> str:
    """读取文件，带大小限制"""
    size = path.stat().st_size
    if size > max_bytes:
        content = path.read_text(errors="replace")[: max_bytes // 2]
        return f"[File too large ({size} bytes, showing first {max_bytes // 2} bytes)]\n{content}"
    return path.read_text(errors="replace")
```

**沙箱机制**：`_resolve_path()` 通过 `relative_to()` 验证解析后的路径仍然在 `workdir` 内。如果用户（或 LLM）尝试读取 `../../etc/passwd`，会被拒绝：

```python
>>> Path("/etc/passwd").relative_to(Path("/home/user/project"))
ValueError: '/etc/passwd' is not in the subpath of '/home/user/project'
```

### 3.2 ToolRegistry（`tools/registry.py`）

所有工具的中央注册表，功能很简单但很关键：

```python
class ToolRegistry:
    def register(self, tool: Tool) -> None     # 注册工具
    def get(self, name: str) -> Tool | None    # 按名称查找
    def get_all(self) -> list[Tool]            # 获取所有工具
    def get_tool_definitions(self) -> list[dict]  # 生成 LLM 需要的 JSON Schema 列表
```

**`get_tool_definitions()` 是关键方法**——它遍历所有已注册工具，调用各自的 `to_llm_schema()`，生成 LLM API 需要的完整工具列表。这个方法在主循环的每一轮都被调用（`master.py:212`）：

```python
tools = self.registry.get_tool_definitions()
```

### 3.3 ToolResult 统一返回格式

定义在 `models.py:74-77`：

```python
class ToolResult(BaseModel):
    content: str                    # 工具返回的文本内容
    is_error: bool = False          # 是否为错误结果
    metadata: dict[str, Any] = {}   # 额外元数据（行数、退出码等）
```

所有工具的 `execute()` 方法都返回这个统一格式，MasterAgent 处理时不需要关心具体工具的返回结构。

## 四、内置工具逐个解析

### 4.1 ReadTool — 读取文件

```python
parameters = {
    "file_path":    {"type": "string",  "required": True},
    "start_line":   {"type": "integer", "required": False},  # 支持行范围
    "end_line":     {"type": "integer", "required": False},
}
```

**关键特性**：
- 支持 `start_line` / `end_line` 行范围，避免一次性读取超大文件
- 输出带行号（`f"{i:6d}  {line}"`），方便 LLM 定位
- 空文件返回 `"(empty file)"` 而不是空字符串

### 4.2 WriteTool — 写入文件

```python
parameters = {
    "file_path": {"type": "string", "required": True},
    "content":   {"type": "string", "required": False},
    "mode":      {"type": "string", "required": False},  # overwrite | append | str_replace
    "old_str":   {"type": "string", "required": False},  # str_replace 模式的旧字符串
    "new_str":   {"type": "string", "required": False},  # str_replace 模式的新字符串
}
```

**三种写入模式**：

| 模式 | 行为 | 举例 |
|------|------|------|
| **overwrite** | 创建或覆盖整个文件 | 创建新文件、完全重写 |
| **append** | 追加到文件末尾 | 添加日志、追加内容 |
| **str_replace** | 精确替换文件中的特定字符串 | 修改变量名、修复 bug |

**str_replace 的安全检查**（`tools/builtin/write.py:53-68`）：
```python
if old_str not in original:
    return ToolResult(
        content=f"Error: 'old_str' not found in file. Cannot perform str_replace.",
        is_error=True,
    )
updated = original.replace(old_str, new_str, 1)  # 只替换第一次出现
```

**为什么只替换一次？** 防止 `old_str` 在文件中多次出现时意外修改了不该修改的地方。这也要求 LLM 提供的 `old_str` 足够唯一（包含足够的上下文）。

### 4.3 BashTool — 命令执行

这是安全控制最严格的工具，有 **4 层防护**：

#### 第 1 层：危险模式检测（`bash.py:46-52`）

```python
def _check_dangerous(self, command: str) -> str | None:
    cmd_lower = command.lower()
    for pattern in self.config.dangerous_patterns:
        if pattern.lower() in cmd_lower:
            return f"Dangerous pattern detected: {pattern}"
    return None
```

默认危险模式（`config.py:13-15`）：
```python
dangerous_patterns = ["rm -rf /", "sudo", ":(){:|:&};:"]  # 最后一个叫 fork bomb
```

**绕过方式**：LLM 可以传 `dangerous=True`，但这需要用户权限批准（`Task` 的权限是 "ask"）。

#### 第 2 层：沙箱检查（`bash.py:54-59`）

```python
_PATH_ESCAPE = re.compile(r"\.\./|\.\.\\")

def _check_sandbox(self, command: str) -> str | None:
    if self._PATH_ESCAPE.search(command):
        return f"Path traversal detected: command escapes workdir"
```

检测命令中是否包含 `../` 或 `..\`，防止 LLM 通过 shell 命令跳出工作目录。

#### 第 3 层：输出截断（`bash.py:61-80`）

```python
def _smart_truncate(self, output: str) -> str:
    """截断输出，保留头部和尾部以保持上下文"""
    max_bytes = self.config.max_output_bytes  # 默认 50000
    if len(output.encode()) <= max_bytes:
        return output

    head_bytes = max_bytes // 3
    tail_bytes = max_bytes // 3
    # ... 头部 + 省略 + 尾部
```

**为什么保留尾部？** 很多命令的错误信息在尾部（如测试框架的总结），截断时保留尾部能让 LLM 看到关键信息。

#### 第 4 层：只读结果缓存（`bash.py:106-112`）

```python
def _cache_key(self, command: str) -> str:
    return hashlib.md5(command.encode()).hexdigest()

# 执行前检查缓存
cache_key = self._cache_key(command)
read_only = all(
    not command.startswith(cmd)
    for cmd in ("git", "make", "python", "pip", "npm", "cargo", "cmake")
)
if read_only and cache_key in self._cache:
    return self._cache[cache_key]
```

**缓存逻辑解读**：
- 只对以 `git`、`make`、`python` 等开头的命令缓存
- **注意**：这里的逻辑是反的！`read_only = all(not ...)` 意味着**不以这些命令开头**的才被认为是只读的
- 实际上这是一个 **bug**：`git status` 和 `ls` 都会被认为不是只读（因为 `ls` 不在排除列表中），而 `git status` 会被缓存

**缓存的潜在问题**：如果 LLM 执行 `ls` 两次，第二次拿到的是缓存结果，可能无法反映文件系统的最新状态。

### 4.4 GlobTool — 文件模式匹配

```python
async def execute(self, pattern: str) -> ToolResult:
    matches = sorted(self.workdir.glob(pattern))
    paths = [str(p.relative_to(self.workdir)) for p in matches if p.is_file()]
```

直接调用 Python 的 `pathlib.Path.glob()`，支持 `**/*.py` 这样的递归模式。限制 100 条防止输出过大。

### 4.5 GrepTool — 文件内容搜索

```python
async def execute(self, pattern, path=".", file_pattern=None):
    regex = re.compile(pattern)
    for file in files:
        content = self._safe_read(file, max_bytes=100_000)
        for i, line in enumerate(content.splitlines(), 1):
            if regex.search(line):

```

**关键特性**：
- 支持正则表达式（`re.compile(pattern)`）
- 可选文件过滤（`file_pattern` 参数配合 glob）
- 输出格式模仿 `grep -n`：`文件:行号: 内容`
- 每个文件读取上限 100KB，结果限制 100 条

## 七、设计权衡与已知问题

### 7.1 工具权限 vs 工具过滤

NexusAgent 有**两套**工具访问控制机制：

| 机制 | 层级 | 作用 |
|------|------|------|
| **权限系统**（PermissionGate） | 运行时 | 用户确认/拒绝每个工具调用 |
| **工具过滤**（SubAgent） | 创建时 | 子 Agent 只获得部分工具，LLM 根本看不到其他工具 |

**关键区别**：权限系统在 LLM 请求执行工具时拦截；工具过滤让 LLM 从 schema 层面就不知道某些工具的存在。

### 7.2 BashTool 缓存逻辑的 Bug

`bash.py:107-112` 的只读检测逻辑反了：

```python
read_only = all(
    not command.startswith(cmd)
    for cmd in ("git", "make", "python", "pip", "npm", "cargo", "cmake")
)
```

- `command = "ls -la"` → `not "ls".startswith("git")` = True → `all` = True → **被认为是只读**
- `command = "git status"` → `not "git".startswith("git")` = False → `all` = False → **不被缓存**

实际效果是：**非构建/解释器命令被缓存，构建/解释器命令不被缓存**。这与预期相反。修复方法是把 `not` 去掉，或者改用 `any`。

### 7.3 WriteTool 的 append 模式实际是覆盖

`write.py:70-73`：

```python
elif mode == "append":
    path.parent.mkdir(parents=True, exist_ok=True)
    path.write_text(content, encoding="utf-8")  # write_text 是覆盖，不是追加！
```

**Bug**：`write_text()` 会覆盖整个文件内容，而不是追加。正确的追加应该用 `open(path, "a").write(content)`。这意味着 `append` 模式的行为与 `overwrite` 完全一样。

### 7.4 工具注册表的线程安全

`ToolRegistry` 使用普通的 `dict` 存储，没有锁保护。在主 Agent 和子 Agent 共享同一个注册表实例时，如果子 Agent 同时调用 `registry.get()`，理论上可能存在竞态条件。但实际上因为 Python 的 GIL 和 dict 操作的原子性，这种竞态在实践中不太可能触发。
